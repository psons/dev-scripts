# The `do.md` File: A Task-Specific Scratchpad

The `do.md` file serves as a short-lived, task-specific "scratchpad" for the current unit of work, managed by the `dtask` command. Its primary purpose is to guide the development and documentation of a single feature or task within its own git branch.

Here’s a breakdown of its key functions:

1.  **Task Decomposition**: It holds a list of small, "microsected" sub-tasks that break down the larger task into a detailed, actionable plan. This is described in `docs/dev/spec/usecases/dtask-and-do-file-tasks.md` as a place for granular steps that guide implementation.

2.  **Git Workflow Management**: The YAML frontmatter in `do.md` is central to the `dtask` workflow. It stores critical metadata used for managing commits and branches, as specified in `docs/dev/spec/dtask-spec.md`:
    *   `workBranch`: The name of the git branch for the current task.
    *   `intendedCommitMessage`: The planned commit message at the start of the task.
    *   `actualCommitMessage`: The final commit message, which can be updated as work progresses and understanding evolves.
    *   `workHeadline`: A concise summary of work, often generated by the `wsum` tool, which can be used as the commit message for intermediate commits.

3.  **Work Summary Log**: As work is committed using `dtask commit --wsum`, a running log of activities is appended under the `# Work Summary` section. This creates a chronological record of the changes made during the task's lifecycle.

4.  **Ephemeral Nature**: The `do.md` file is designed to be temporary. It is created at the beginning of a task with `dtask init` and removed at the end with `dtask commit --final`. This ensures that the working directory is clean and ready for the next task, and that only incomplete tasks are kept in the backlog.   The backlog is managed using backlog.py and kept by default in the `TODO.md` file.

5.  **History in Git**: because the old do.md files are kept in a detailed branch, the detailed work history can be assembled using git commands to mine the details out of the commit history.

In essence, `do.md` is the "single source of truth" for the immediate task at hand, combining planning, metadata, and progress logging into one convenient, temporary file that drives the `dtask` development cycle.
